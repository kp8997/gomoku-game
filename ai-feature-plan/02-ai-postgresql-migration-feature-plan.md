## 1. Objective

Replace the SQLite file-based database with a PostgreSQL instance to support proper concurrency, production-grade persistence, and scalable infrastructure. All existing JPA entities, repositories, and auth logic (JWT, signup, login, profile, confrontation records) must remain fully functional with zero frontend changes.

**Backward Compatibility**: Anonymous play is unaffected. Authenticated user workflows (signup â†’ login â†’ JWT â†’ profile â†’ H2H) are unchanged at the API level.

**Removed**: `spring.datasource.hikari.maximum-pool-size=1` (SQLite single-writer workaround).

### 3.3 JPA Entity Fix â€” `User.java` (CRITICAL)

**Problem**: The `avatar` field uses `@Lob` annotation. With SQLite, `@Lob` on a `String` mapped to `TEXT` naturally. With PostgreSQL + Hibernate, `@Lob` on `String` maps to `oid` (binary large object reference), which causes data corruption or `PSQLException` at runtime.

**Fix**: Replace `@Lob` with `@Column(columnDefinition = "TEXT")` on the `avatar` field.

```diff
-    @Lob
-    @Column(name = "avatar")
+    @Column(name = "avatar", columnDefinition = "TEXT")
     private String avatar; // Base64 encoded image string
```

### 3.4 ID Generation Strategy

`GenerationType.IDENTITY` is used by both entities. PostgreSQL supports this natively via `SERIAL`/`BIGSERIAL` columns. âś… No change needed.

### 3.5 JPQL Queries

All queries in `ConfrontationRepository.java` use standard JPQL (no native SQL, no SQLite-specific syntax). âś… No change needed.

## 7. Docker Deployment

### 7.1 New Service: `postgres`

```yaml
postgres:
  image: postgres:16-alpine
  container_name: gomoku-postgres
  ports:
    - "5434:5432"
  environment:
    - POSTGRES_DB=gomoku_db
    - POSTGRES_USER=admin-gomoku
    - POSTGRES_PASSWORD=gomoku_password
  volumes:
    - gomoku-pg-data:/var/lib/postgresql/data
  healthcheck:
    test: ["CMD-SHELL", "pg_isready -U admin-gomoku -d gomoku_db"]
    interval: 5s
    timeout: 5s
    retries: 5
  networks:
    - gomoku-network
```

### 7.2 Updated `backend` Service

```yaml
backend:
  # ... existing build config ...
  environment:
    - DB_HOST=postgres
    - DB_PORT=5432
    - DB_NAME=gomoku_db
    - DB_USER=admin-gomoku
    - DB_PASSWORD=gomoku_password
  depends_on:
    postgres:
      condition: service_healthy
```

**Removed**: `volumes: gomoku-db:/app/data` (SQLite volume).  
**Removed**: `gomoku-db` from top-level `volumes:`.  
**Added**: `gomoku-pg-data` to top-level `volumes:`.

### 7.3 CI/CD Workflow Update (`.github/workflows/deploy.yml`)

The deploy script currently only rebuilds `backend`:
```bash
docker compose up -d --build backend
```

After migration, the `postgres` service must also be running. Update to:
```bash
docker compose up -d --build postgres backend
```

This ensures PostgreSQL is started (or already running) before the backend container starts.

### 7.4 Local Development (Without Docker)

For running the Spring Boot backend directly (not via Docker), PostgreSQL must be accessible on `localhost:5432` with the configured credentials, or start a standalone PostgreSQL container:

```bash
docker run -d --name gomoku-pg-dev \
  -e POSTGRES_DB=gomoku_db \
  -e POSTGRES_USER=admin-gomoku \
  -e POSTGRES_PASSWORD=gomoku_password \
  -p 5434:5432 \
  postgres:16-alpine
```

Then run Spring Boot with:
```bash
DB_HOST=localhost DB_PORT=5434 ./mvnw spring-boot:run
```
